# Terraform Module Versioning (v4)

## Overview

All Terraform modules must follow semantic versioning (SemVer) and be pinned to specific versions. Version 4 introduces mandatory module signing and provenance attestation.

## Requirements

### Version Format
- Use SemVer: `MAJOR.MINOR.PATCH`
- Tag releases in the module repository
- Document breaking changes in CHANGELOG.md
- **NEW in v4:** All tags must be signed with a GPG key registered in the module registry

### Pinning Rules
- Always pin to exact versions in production configurations
- Use version constraints (`~>`) only in development environments
- Never use `ref = "main"` in module sources
- **NEW in v4:** Version pins must include a SHA256 integrity hash in `terraform.lock.hcl`

### Module Registry
- All shared modules must be published to the internal Terraform module registry
- Module names must follow the pattern: `terraform-<PROVIDER>-<NAME>`
- Each module must include a `versions.tf` file declaring required provider versions
- **NEW in v4:** Modules must include a `provenance.json` attestation file

### Provenance Attestation (NEW in v4)
- Every published module version must include a SLSA Level 2 provenance attestation
- The attestation must be generated by the CI/CD pipeline, not manually
- Consumers can verify provenance using `terraform providers lock -verify`

## Examples

### Correct Usage (v4)
```hcl
module "network" {
  source  = "app.terraform.io/myorg/network/azurerm"
  version = "4.0.0"
}
```

With integrity hash in `terraform.lock.hcl`:
```hcl
provider "registry.terraform.io/myorg/azurerm" {
  version     = "4.0.0"
  constraints = "4.0.0"
  hashes = [
    "h1:abc123...",
  ]
}
```

## Migration from v3

- Add GPG key to module registry settings
- Generate `provenance.json` in your CI/CD pipeline
- Update `terraform.lock.hcl` with integrity hashes: `terraform providers lock`
- No changes required to existing module source references
