Fig 1. System topology — Gateway orchestrates Index + Standards, all backed by Postgres + Ollama

Gateway :8080

MCP protocol endpoint. Orchestrates context bundle assembly, enforces policy, caches bundles. The single entry point for all AI clients.

Index Service :8081

Ingests memory packs, computes embeddings, provides semantic search. Manages the provider framework (GitHub, Filesystem) and repo registry.

Standards Service :8082

Serves enterprise standards live from GitHub. Two-layer caching (ETag + blob SHA) minimizes API calls. Updates propagate automatically.

PostgreSQL + pgvector Data

Vector database with HNSW indexing for cosine similarity search. Stores chunks, embeddings, packs, caches, and the repo registry.

Ollama Data

Local embedding model (nomic-embed-text, 768 dimensions). Generates embeddings for both indexing and query-time search.

GitHub API External

Source of truth for repo memory packs and enterprise standards. ETag-based caching keeps steady-state costs at 0–2 API calls per repo.

Key Insight: Zero-Friction After Setup

Once a repo is registered and indexed, add one line to CLAUDE.md:

Before starting any coding task, call get_context_bundle with the repo name and task description.

From that point, every AI-assisted task automatically gets the right enterprise standards — developers never have to ask.

Fig 6. Three caching layers minimize latency and GitHub API usage

Layer 1 Bundle Cache

What: Entire assembled bundles (chunks + standards + metadata)

Key: SHA-256 of namespace:repo:task_hash:ref:version

TTL: 300 seconds (configurable)

Effect: Repeated identical tasks return instantly

Layer 2 GitHub Tree Cache

What: Git tree structure (file paths + blob SHAs)

Key: ETag from GitHub API

TTL: Until tree changes (ETag mismatch)

Effect: Steady-state indexing = 1 API call (304 response)

Layer 3 Blob Content Cache

What: Raw file content keyed by Git blob SHA

Key: Blob SHA (content-addressable, immutable)

TTL: Forever (SHA changes when content changes)

Effect: Changed files re-fetch; unchanged files never re-fetch

9. Security Model

✓
Repo validation before registration — GitHub API confirms the repo exists and has .ai/memory/manifest.yaml before it's stored. No phantom or typosquatted repos.
✓
Env-var repo protection — Repos configured via GITHUB_REPOS cannot be unregistered via MCP. Only DB-registered repos can be removed dynamically.
✓
Internal service auth — All gateway-to-service communication uses X-Internal-Token headers. New endpoints use the same guard.
✓
No code execution — Memory packs contain markdown text only. Content is chunked, embedded, and stored — never evaluated, rendered, or executed.
✓
Registration rate limiting — MAX_REGISTERED_REPOS cap (default 50) prevents mass registration. Configurable via env var.
✓
Namespace isolation — All queries filter by namespace. Repos in team-a namespace are invisible to team-b.
✓
No credentials in MCP API — The register_repo tool accepts owner/repo only. GitHub auth uses the pre-configured server-side token.
✓
Classification-based filtering — All content is tagged internal. The classification system ensures corporate context stays within the organization's AI tooling.

Data Classification

All content is classified internal — this is a corporate-internal system providing context to AI assistants working within the organization. All indexed repos and enterprise standards are internal corporate assets.

Audit Trail

Every tool call is logged with:

  • Tool name + action (invoke/deny/error)
  • Subject (role), repo, correlation ID
  • Duration (ms), provenance refs
